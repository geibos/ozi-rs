# Feature Status

This table separates backend capability, UI exposure, documentation state, implementation
status, and evidence. Use it to avoid broad "supported" claims when a feature is only
partially available.

| Feature | Backend | UI | Docs | Status | Evidence |
|---------|---------|----|------|--------|----------|
| Startup restore | Infrastructure session persistence stores the last project path and active map reference/path separately from `.ozp` files. Missing referenced files degrade to fresh state or restored project with a warning instead of panic. | Startup uses the restored project/map only. No viewport, panel, selected entity, undo, theme, unsaved edit, or bundle-loader window state is restored by the Rust session file. | README, roadmap, architecture, ADR-0019, and `docs/persistence-session.md` describe bounded restore scope. | Implemented, bounded | Task 2 session restore tests and notepad evidence. |
| Track point timestamp display | Imported/project track points expose optional timestamp values through track-detail DTOs. | `TrackPointsPanel.svelte` renders timestamp text when present and omits placeholder text when absent. Sorting by timestamp remains planned/not implemented. | Requirements and roadmap distinguish timestamp display from deferred timestamp sorting. | UI surfaces existing data | Task 3 `src/test/track-points-panel.test.ts` source-level guard. |
| Track color controls | Backend supports per-track color mutation and GeoJSON rendering uses color properties. | `TracksPanel.svelte` surfaces a color picker that calls the typed `setTrackColor()` wrapper. | README, requirements, frontend architecture, and this matrix list color as surfaced UI. | Implemented UI control | Task 4 `src/test/tracks-panel-style.test.ts`; no direct component `invoke()` guard. |
| Track line width controls | Backend supports per-track line-width mutation and GeoJSON rendering uses width properties. | `TracksPanel.svelte` surfaces a 1-12 px line-width control that calls the typed `setTrackLineWidth()` wrapper. | README, requirements, frontend architecture, and this matrix list line width as surfaced UI. | Implemented UI control | Task 4 `src/test/tracks-panel-style.test.ts`. |
| Waypoint symbol undo | `ProjectCommand::SetWaypointSymbol` routes symbol changes through the command stack with old/new symbols for reverse deltas. | Existing symbol picker commits choices through `setWaypointSymbol()`; undo/redo applies to each committed symbol choice. | Architecture and commands reference list `SetWaypointSymbol`/`set_waypoint_symbol` as undoable. | Undoable project edit | Task 5 `waypoint_symbol_undo` and `waypoint_symbol_missing` Rust tests. |
| OK-standard warning | Backend remains permissive; it does not reject non-conforming names. | `TracksPanel.svelte` surfaces warning-only validation for `^\d{8}_.*\S.*$`, allowing Cyrillic callsigns and not validating calendar dates. Rename/save/export are not blocked. | README, requirements, roadmap, ADR-0019, and this matrix use warning-only language. | UI warning implemented | Task 6 `src/test/track-name-validation.test.ts`. |
| 10-Tracks export default | Backend capability `get_track_export_default_path` builds active-bundle `10-Tracks/<track>.<ext>` suggestions and returns no suggestion when no active bundle can be inferred. | GPX and PLT export dialogs use the backend suggestion when present and keep filename-only fallback otherwise. Users can choose another path. | README, requirements, roadmap, commands reference, ADR-0019, and this matrix describe suggested defaults, not forced export. | Implemented suggestion | Task 7 `export_default_tracks_dir` Rust test and `src/test/export-default-path.test.ts`. |
| Multi-layer workflows | Backend supports multiple track and waypoint layers and `AppStateDto` exposes layer summaries. | UI surfaces active-layer selection for existing drawing, waypoint add/move/list, and track-detail workflows. Full layer management (create/rename/delete/reorder UI) is not implemented. | README, requirements, roadmap, frontend architecture, ADR-0019, and this matrix say active-layer selection rather than full layer management. | Minimal UI implemented; full manager planned/not implemented | Task 8 `src/test/active-layer-ui.test.ts` and no-hardcoded-workflow guard. |
| `windows.ts` bundle-loader lifecycle | Tauri secondary window support exists through a `bundles` webview at `/?view=bundles`. | `src/main.ts` pre-creates the hidden loader on main-window startup; `Sidebar.svelte` opens it through `openBundleLoader()`, which reuses, shows, and focuses the existing window or creates a fallback. Session restore does not persist window open/closed state. | `docs/frontend-architecture.md` documents `src/lib/windows.ts`, `precreateBundleLoader()`, `openBundleLoader()`, and the `view=bundles` route; `docs/persistence-session.md` states window open state is not restored. | Documented current behavior | Task 9 required-term check and window-session grep. |
| Backend capability/status split | Backend capability varies by feature: some behaviors are fully implemented, some are backend-only, and some remain planned/not implemented. | UI exposure is listed separately so docs do not equate backend support with surfaced workflows. | This table is the public status split for audit findings and uses “backend supports,” “UI surfaces,” “planned,” and “not implemented” where appropriate. | Complete for the 10 audit findings | Task 10 feature-status coverage and unsupported-claims Python checks. |
