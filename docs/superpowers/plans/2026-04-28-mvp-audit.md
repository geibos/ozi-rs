# MVP Audit Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce a verified status of every MVP feature line from ADR-0020 through driven QA via `tools/ozi-rs-mcp`, deliver an updated `docs/feature-status.md`, a triage list, an initial smoke-document set under `docs/qa/`, and one promoted Appium test that exercises the highest-risk smoke.

**Architecture:** Walk the MVP-scope checklist from ADR-0020 in batches grouped by area (maps → tracks → waypoints → on-map tools → project/UI). For each feature, run the verification protocol from `docs/agent-verification.md`: build → launch → drive Appium-clicked smoke → observe → classify → record. Aggregate findings into a single update to `docs/feature-status.md`, then promote the bundle-open / map-switch smoke (the user's known critical bug) into a real Appium test as proof of the promotion path.

**Tech Stack:** `tools/ozi-rs-mcp` (rmcp + Tier 1 native + Tier 2 Appium Mac2), Tauri 2 + Svelte 5 frontend, real bundle fixture at `example_data/2018-09-26_Nizovskaya/`.

**Primary fixture bundle:** `/Users/sobieg/Projects/ozi-rs/example_data/2018-09-26_Nizovskaya/` — contains:

- `5-Ozi(Win&Android)_Topo_EEKO.zip` and `6-Ozi(Win&Android)_Satell.zip` — unzip for OZF2 `.map`+`.ozf2` raster fixtures

Screen Recording permission. `mcp__ozi-rs-mcp__capture_screenshot` returns

(see `docs/qa/2026-04-29-tooling-audit-findings.md` finding F2). Grant
access to the MCP host (Claude Code or opencode) in *System Settings →
Privacy & Security → Screen Recording* before running the audit, then
restart the MCP client so the grant takes effect.

**If `appium_launch_session` returns `server_unavailable` despite curl
reaching `/status`.** The Mac2 driver can enter a state where new sessions
hang (finding F7). Run `brew services restart appium`, then retry. Do not
loop more than once on this — escalate to the user.

## Task 2: Smoke-document template and shared procedure

**Goal:** Define the per-feature audit procedure once so the rest of the plan can reference it without repetition.

**Files:**
- Create: `docs/qa/_template.md`
- Create: `docs/qa/_audit-procedure.md`

- [ ] **Step 2.1: Write the smoke-document template**

Create `docs/qa/_template.md` with this exact content:

```markdown
# Smoke: <feature-name>

Source: ADR-0020 (MVP scope), section <area>.

## Preconditions
- App built (Task 1, this plan)
- App launched via `launch_app`
- Fixtures: <list .gpx / .plt / bundle paths used>

## UI entry point
- Selector: <Appium accessibility selector or coords with screenshot>
- Notes: <how to reach this entry point from app start>

## Steps and expected outcomes

1. **Action**: <Appium call, e.g. `appium_click(selector="//button[@title='Maps']")`>
   **Expected**: <observable property — text appears, list non-empty, panel opens>
   **Artifact**: <screenshot path>

2. **Action**: ...
   **Expected**: ...
   **Artifact**: ...

## Classification
- [ ] works (all steps pass)
- [ ] partial (happy path passes; documented edge case fails — see Known failure modes)
- [ ] broken (UI entry reachable, but action does not produce expected outcome)
- [ ] hidden (no UI entry point found; backend may exist)
- [ ] missing (neither UI nor backend)

## Evidence
- Build: <evidence path>
- Launch: <evidence path>
- Action screenshots: <list>

## Known failure modes
- <pattern> → <likely cause> → <next diagnostic step>
```

- [ ] **Step 2.2: Write the audit procedure**

Create `docs/qa/_audit-procedure.md` with this exact content:

```markdown
# Audit Procedure (per feature)

Apply this procedure to every MVP-scope feature listed in ADR-0020.

1. Open or create `docs/qa/smoke-<feature>.md` from `_template.md`.
2. Identify the UI entry point: open accessibility inspector or query Appium
   for the element. Record the selector in the smoke doc.
3. `build_app` (skip if a fresh build from this session already exists).
4. `launch_app`.
5. `qa_observe` to record the baseline screenshot for this feature.
6. For each step in the smoke doc:
   - Drive the action via the appropriate Appium tool.
   - `appium_screenshot` (or `qa_observe`) immediately after.
   - Compare against the smoke doc's "expected outcome" and record actual.
7. Classify the feature as works / partial / broken / hidden / missing.
8. `stop_app`.
9. Append the row to the in-progress triage list in `docs/qa/triage.md`.

If any step fails to produce an expected outcome, do not retry the action
twice. Record the failure as Classification = broken (or partial), capture
the artefact, and move on. Diagnosis happens in the post-audit triage,
not inside the audit itself.
```

- [ ] **Step 2.3: Commit the procedure**

```bash
git add docs/qa/_template.md docs/qa/_audit-procedure.md
git commit -m "qa: add audit procedure and smoke-document template

Single source for the per-feature audit walk used by the remaining tasks
in docs/superpowers/plans/2026-04-28-mvp-audit.md."
```
